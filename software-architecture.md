## 一、宏观分层架构(系统级)

### 1.1 单体架构(Monolithic Architecture)

单体架构是最原始也最直观的架构形式:整个应用被打包成一个可部署单元,所有功能模块(用户、订单、支付、通知)运行在同一个进程中,共享同一个数据库。

**优势:** 开发简单、调试方便、部署成本低、模块间调用无网络开销,适合团队规模小(< 10 人)、业务初期快速验证的场景。

**劣势:** 随着业务增长,代码库膨胀导致编译慢、部署风险高(改一行代码需全量发布)、技术栈无法灵活选择、水平扩展只能整体扩展而无法针对热点模块。

```plaintext
┌─────────────────────────────────────┐
│          单体应用进程               │
│  ┌──────┐  ┌──────┐  ┌──────────┐  │
│  │用户模块│  │订单模块│  │支付模块 │  │
│  └──────┘  └──────┘  └──────────┘  │
│  ┌──────┐  ┌──────┐  ┌──────────┐  │
│  │商品模块│  │库存模块│  │通知模块 │  │
│  └──────┘  └──────┘  └──────────┘  │
└──────────────────┬──────────────────┘
                   │
            ┌──────▼──────┐
            │  单一数据库  │
            └─────────────┘
```

**适用场景:** 创业初期、内部工具、业务逻辑简单的中小型应用。

---

### 1.2 分层架构(Layered Architecture)

分层架构是单体架构的进化,将系统按职责划分为水平层次,每层只与相邻层交互,是企业应用最经典的组织方式。

**经典四层模型:**

```plaintext
┌─────────────────────────────┐
│  表现层(Presentation)       │  ← 处理 HTTP 请求,返回响应
├─────────────────────────────┤
│  业务层(Business)           │  ← 核心业务逻辑,不依赖框架
├─────────────────────────────┤
│  持久层(Persistence)        │  ← 数据访问,ORM / SQL
├─────────────────────────────┤
│  数据库层(Database)         │  ← MySQL / PostgreSQL 等
└─────────────────────────────┘
```

**关键原则:** 依赖方向只能从上到下,业务层不应知道数据库的存在(通过接口抽象),表现层不应包含业务逻辑。这种关注点分离使各层可以独立测试和替换。

---

### 1.3 微服务架构(Microservices Architecture)

微服务将单体应用拆分为一组**小型、独立部署、围绕业务能力构建**的服务,每个服务拥有自己的进程和数据库,通过轻量级 API(HTTP/gRPC)或消息队列通信。

```plaintext
       API Gateway(统一入口)
              │
      ┌───────┼───────┐
      ▼       ▼       ▼
  ┌───────┐ ┌───────┐ ┌───────┐
  │用户服务│ │订单服务│ │支付服务│
  │  DB   │ │  DB   │ │  DB   │
  └───────┘ └───────┘ └───────┘
      │         │
      └────┬────┘
           ▼
    消息队列(Kafka)
           │
      ┌────┴────┐
      ▼         ▼
  ┌───────┐ ┌───────┐
  │库存服务│ │通知服务│
  │  DB   │ │  DB   │
  └───────┘ └───────┘
```

**核心优势:**
- 独立部署:修改订单服务不影响用户服务
- 技术异构:订单服务用 Java,推荐服务用 Python
- 弹性扩展:只扩展热点服务(如大促时扩展订单服务)
- 故障隔离:一个服务崩溃不会拖垮整个系统

**核心挑战:**
- 分布式系统复杂性(网络延迟、部分失败)
- 数据一致性难以保证(跨服务事务)
- 运维成本高(需要服务发现、链路追踪、日志聚合)
- 服务间调用的序列化/反序列化开销

**适用场景:** 团队规模大(> 50 人)、业务复杂、需要高可用和弹性扩展的互联网产品。

---

### 1.4 服务网格架构(Service Mesh)

服务网格是微服务架构的进化,将服务间通信的横切关注点(负载均衡、熔断、重试、链路追踪、mTLS 加密)从业务代码中剥离,下沉到基础设施层(Sidecar 代理)。

```plaintext
┌──────────────────────────────────────┐
│              服务 A                   │
│  ┌──────────┐  ┌──────────────┐      │
│  │  业务代码 │◄──►│ Sidecar Proxy│      │
│  └──────────┘  └──────┬───────┘      │
└─────────────────────────┼────────────┘
                          │ mTLS
┌─────────────────────────┼────────────┐
│              服务 B      │            │
│  ┌──────────┐  ┌──────▼───────┐      │
│  │  业务代码 │◄──►│ Sidecar Proxy│      │
│  └──────────┘  └──────────────┘      │
└──────────────────────────────────────┘

  控制平面(Istio / Linkerd)统一管理所有 Sidecar
```

Istio 是目前最主流的服务网格实现,业务开发者无需关心服务治理细节,专注于业务逻辑本身。

---

### 1.5 Serverless 架构

Serverless(无服务器)并非没有服务器,而是开发者**无需管理服务器**。代码以函数(Function)为单位部署,由事件触发执行,按实际调用次数和执行时长计费,空闲时不产生费用。

```plaintext
HTTP 请求    ──► API Gateway  ──► Lambda 函数  ──► DynamoDB
定时任务     ──► EventBridge  ──► Lambda 函数  ──► S3
文件上传     ──► S3 事件      ──► Lambda 函数  ──► SES(发邮件)
```

**优势:** 零运维、自动弹性伸缩、按量付费、极低的冷启动成本。\
**劣势:** 冷启动延迟、执行时间限制(AWS Lambda 最长 15 分钟)、本地调试困难、厂商锁定风险。

**适用场景:** 事件驱动的异步任务、流量极不规律的 API、定时任务、数据处理管道。

## 二、前端架构

### 2.1 MPA vs SPA 的根本区别

```plaintext
MPA(多页应用)                    SPA(单页应用)
每次导航 → 服务器返回新 HTML     首次加载 → 下载 JS Bundle
浏览器完整刷新                   后续导航 → JS 动态更新 DOM
SEO 天然友好                     SEO 需要额外处理
首屏快,后续导航慢                首屏慢,后续导航极快
```

---

### 2.2 SSR / SSG / ISR 渲染模式对比

| 模式 | 渲染时机 | 数据新鲜度 | 适用场景 |
|---|---|---|---|
| **CSR**(客户端渲染) | 浏览器运行时 | 实时 | 后台管理系统、登录后页面 |
| **SSR**(服务端渲染) | 每次请求时 | 实时 | 新闻、电商详情页 |
| **SSG**(静态生成) | 构建时 | 构建时快照 | 博客、文档、营销页 |
| **ISR**(增量静态再生) | 构建时 + 定时重新生成 | 准实时 | 电商列表页、内容平台 |

Next.js 将这四种模式统一在同一框架中,可以按页面粒度选择渲染策略,是目前最灵活的前端框架。

---

### 2.3 微前端架构(Micro-Frontend)

微前端将微服务思想引入前端,允许多个独立团队各自开发、部署自己的前端模块,在运行时组合成一个完整应用。

```plaintext
┌──────────────────────────────────────────┐
│              主应用(Shell)                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │ 子应用A  │  │ 子应用B  │  │ 子应用C  │ │
│  │(React团队)│  │(Vue团队) │  │(Angular) │ │
│  └──────────┘  └──────────┘  └──────────┘ │
└──────────────────────────────────────────┘
```

**实现方案:**
- **Module Federation(Webpack 5)**:目前最主流,模块在运行时动态共享
- **iframe**:隔离性最强,但体验最差
- **Web Components**:标准化方案,框架无关
- **qiankun / single-spa**:国内常用的微前端框架

## 三、后端架构模式

### 3.1 MVC 架构

MVC(Model-View-Controller)是后端最经典的架构模式,将应用分为三个角色:

```plaintext
        用户请求
            │
            ▼
Controller(控制器)── 调用 ──► Model(模型/数据层)
            │                      │
            │ 选择视图             │ 返回数据
            ▼                      │
        View(视图)◄─────────────────┘
            │
            ▼
      返回响应给用户
```

- **Model**:数据和业务逻辑,与数据库交互
- **View**:展示层,渲染 HTML 模板(服务端渲染场景)或 JSON(API 场景)
- **Controller**:接收请求,协调 Model 和 View

Rails、Django、Laravel、Spring MVC 都是 MVC 的经典实现。

---

### 3.2 领域驱动设计(DDD)

DDD 是应对复杂业务系统的架构方法论,核心思想是让**软件模型与业务领域模型保持一致**。

**核心概念:**
- **限界上下文(Bounded Context)**:将大型业务领域划分为若干子域,每个子域有自己独立的模型和语言。例如"用户"在"订单上下文"中是 `Customer`,在"权限上下文"中是 `Principal`。
- **聚合根(Aggregate Root)**:一组相关对象的入口,保证业务不变量。例如 `Order` 是聚合根,`OrderItem` 只能通过 `Order` 操作。
- **领域事件(Domain Event)**:业务中发生的重要事情,如 `OrderPlaced`、`PaymentCompleted`,驱动跨上下文的业务流程。
- **仓储(Repository)**:抽象数据访问,领域层不知道数据库的存在。

```plaintext
┌──────────────────────────────────────────┐
│          订单限界上下文                   │
│  ┌──────────────────────────────────┐    │
│  │      Order(聚合根)                │    │
│  │      ├── OrderItem(值对象)       │    │
│  │      ├── ShippingAddress(值对象) │    │
│  │      └── OrderStatus(枚举)       │    │
│  └──────────────────────────────────┘    │
│  领域事件: OrderPlaced → 触发库存扣减    │
└──────────────────────────────────────────┘
```

---

### 3.3 六边形架构(Hexagonal Architecture)

又称"端口与适配器架构",核心思想是**业务逻辑与外部世界完全隔离**。业务核心(六边形内部)通过"端口"(接口)与外部交互,"适配器"负责将外部技术(HTTP、数据库、消息队列)转换为业务能够理解的形式。

```plaintext
HTTP 适配器              │              CLI 适配器
                         ▼       消息队列适配器
                    ┌────┴────┐
                    │         │
                    │ 业务核心 │
                    │ (纯逻辑) │
                    │         │
                    └────┬────┘
                         │
        DB 适配器        │    外部 API 适配器
```

这种架构使业务逻辑可以在没有数据库、没有 HTTP 框架的情况下被独立测试,是高可测试性架构的典范。

---

### 3.4 CQRS(命令查询职责分离)

CQRS 将系统操作分为两类:**Command(写操作,改变状态)** 和 **Query(读操作,不改变状态)**,并为它们建立独立的模型和数据存储。

```plaintext
         用户操作
            │
├── 写操作(Command)──► 命令处理器 ──► 写库(规范化,保证一致性)
│           │
│      领域事件 ──► 投影(Projection)
│           │
└── 读操作(Query)──► 查询处理器 ──► 读库(反规范化,专为查询优化)
```

**典型场景:** 电商系统中,下单写入规范化的关系型数据库,而商品列表查询则读取专门为展示优化的 Elasticsearch 索引,两者完全独立,互不影响。

---

### 3.5 事件溯源(Event Sourcing)

事件溯源不存储当前状态,而是**存储所有导致当前状态的事件序列**,当前状态通过重放事件得出。

```plaintext
传统方式:存储最终状态
users 表: { id: 1, balance: 300 }

事件溯源:存储事件流
events:
  1. AccountOpened { balance: 0 }
  2. MoneyDeposited { amount: 500 }
  3. MoneyWithdrawn { amount: 200 }
  → 重放得出: balance = 0 + 500 - 200 = 300
```

**优势:** 完整的审计日志、可以回溯到任意历史时刻、天然与 CQRS 结合。\
**劣势:** 查询当前状态需要重放事件(需要快照优化)、事件模型设计复杂。

### 4.2 缓存架构

缓存是提升系统性能的最有效手段,分为多个层次:

```plaintext
        用户请求
            │
            ▼
    CDN 缓存(静态资源,全球边缘节点)
            │ 未命中
            ▼
    Nginx 缓存(反向代理层缓存)
            │ 未命中
            ▼
    应用层缓存(本地内存,如 LRU Cache)
            │ 未命中
            ▼
    Redis 分布式缓存
            │ 未命中
            ▼
        数据库(最终数据源)
```

**缓存策略:**
- **Cache-Aside(旁路缓存)**:应用先查缓存,未命中再查数据库并回填,最常用
- **Write-Through(写穿)**:写操作同时更新缓存和数据库,保证一致性
- **Write-Behind(写回)**:先写缓存,异步批量写数据库,写性能极高但有数据丢失风险

**缓存问题:**
- **缓存穿透**:查询不存在的数据,每次都打到数据库 → 布隆过滤器
- **缓存雪崩**:大量缓存同时过期 → 随机过期时间 + 熔断降级
- **缓存击穿**:热点 Key 过期瞬间大量并发 → 互斥锁重建

## 五、分布式架构

### 5.1 CAP 理论

分布式系统在网络分区(P)发生时,只能在**一致性(C)**和**可用性(A)**之间二选一:

- **CP 系统**(如 ZooKeeper、HBase):网络分区时拒绝服务,保证数据一致性
- **AP 系统**(如 Cassandra、DynamoDB):网络分区时继续服务,但数据可能不一致
- **CA 系统**:只存在于单机,分布式系统中 P 不可避免

实践中更常用 **BASE 理论**(基本可用、软状态、最终一致性)代替强一致性,通过异步补偿机制达到最终一致。

---

### 5.2 消息队列架构

消息队列是分布式系统解耦的核心工具,实现服务间的异步通信:

```plaintext
生产者(Producer)──► Topic/Queue ──► 消费者(Consumer)

应用场景:
订单服务 ──[order.created]──► 库存服务(扣减库存)
                             ──► 积分服务(增加积分)
                             ──► 通知服务(发短信)
                             ──► 数据服务(更新报表)
```

| 消息队列 | 特点 | 适用场景 |
|---|---|---|
| **Kafka** | 高吞吐、持久化、流处理 | 日志收集、事件流、大数据管道 |
| **RabbitMQ** | 功能丰富、路由灵活 | 任务队列、RPC、复杂路由 |
| **RocketMQ** | 阿里系、事务消息 | 电商交易、金融场景 |
| **Redis Pub/Sub** | 轻量、实时 | 简单通知、实时推送 |

---

### 5.3 分布式事务

跨服务的数据一致性是微服务最难解决的问题:

**Saga 模式(推荐):**
将分布式事务拆分为一系列本地事务,每步成功后发布事件触发下一步,失败时执行补偿事务回滚。

```plaintext
下单 Saga:
1. 创建订单(Order Service)
2. 扣减库存(Inventory Service) ← 失败则执行补偿:恢复库存
3. 扣减余额(Payment Service)  ← 失败则执行补偿:退款 + 恢复库存
4. 发送通知(Notification Service)
```

**两阶段提交(2PC):**
强一致性但性能差,协调者单点故障风险高,不适合高并发场景。

## 六、云原生架构

### 6.1 容器化体系

```plaintext
代码 → Dockerfile → Docker 镜像 → 容器实例

Dockerfile 最佳实践:
✓ 使用多阶段构建减小镜像体积
✓ 非 root 用户运行
✓ 利用层缓存优化构建速度
✓ 使用 .dockerignore 排除无关文件
```

### 6.2 Kubernetes 核心架构

```plaintext
Kubernetes 集群
├── Control Plane(控制平面)
│   ├── API Server     ── 所有操作的入口
│   ├── etcd           ── 集群状态存储
│   ├── Scheduler      ── 决定 Pod 调度到哪个节点
│   └── Controller     ── 确保实际状态与期望状态一致
│
└── Worker Nodes(工作节点)
    ├── kubelet        ── 管理节点上的 Pod
    ├── kube-proxy     ── 网络代理,实现 Service 负载均衡
    └── Pods           ── 最小部署单元,包含一个或多个容器
```

**核心资源对象:**
- **Deployment**:声明式管理 Pod 副本数,支持滚动更新和回滚
- **Service**:为一组 Pod 提供稳定的网络访问入口
- **Ingress**:HTTP 路由规则,将外部流量路由到内部 Service
- **ConfigMap / Secret**:配置和敏感信息管理
- **HPA**:根据 CPU/内存自动水平扩缩容

### 6.4 可观测性体系(Observability)

现代系统的可观测性由三大支柱构成:

```plaintext
        可观测性三支柱
Metrics(指标)      Logs(日志)       Traces(链路追踪)
数值型时序数据     文本型事件记录    请求跨服务的完整路径
CPU/内存/QPS/延迟  错误信息/业务日志  定位性能瓶颈
Prometheus+Grafana ELK Stack        Jaeger / Zipkin
```

**告警策略:**
基于 SLO(服务级别目标)设置告警阈值,如"99% 的请求延迟 < 200ms",超出时触发 PagerDuty 通知值班工程师。

## 七、架构选型决策框架

### 如何选择合适的架构?

```plaintext
        业务规模与团队规模
              │
├── 初创期(1-10人,业务未验证)
│   → 单体架构 + 分层设计
│   → 快速迭代,避免过度设计
│
├── 成长期(10-50人,业务快速增长)
│   → 模块化单体 或 少量微服务拆分
│   → 引入消息队列解耦核心链路
│
└── 成熟期(50人+,业务稳定复杂)
    → 微服务 + DDD 领域划分
    → 服务网格 + 完整可观测性
    → 多云 / 混合云部署
```

### 架构演进原则

1. **不要过度设计**:用最简单的架构满足当前需求,架构应该随业务演进而演进
2. **康威定律**:系统架构往往反映团队的沟通结构,先设计团队组织再设计架构
3. **渐进式迁移**:从单体到微服务不是一次性重写,而是逐步抽取服务
4. **可观测性优先**:在系统复杂化之前,先建立完善的监控和日志体系
5. **拥抱失败**:分布式系统中故障是常态,设计时假设任何组件都会失败

## 总结:架构演进路线图

```plaintext
阶段一:单体起步
代码 → 单一进程 → 单一数据库 → 直接部署服务器

阶段二:分层优化
表现层 / 业务层 / 数据层分离 → 引入缓存 → CDN 加速

阶段三:服务化拆分
API Gateway → 核心服务独立 → 消息队列解耦 → 读写分离

阶段四:云原生成熟
容器化 → K8s 编排 → CI/CD → 服务网格 → 完整可观测性

阶段五:平台化演进
多云部署 → 混沌工程 → AIOps → 自愈系统
```

每个阶段都有其适合的业务规模和团队能力要求。**最好的架构,是当下最适合的架构。**
