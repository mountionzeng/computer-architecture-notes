## 1. 思考脉络

1. 有序数组查得快（二分）但插入慢 O(n) → 想结合"快查"+"快改"
2. 把二分过程做成一棵树 → **BST**
3. 中序遍历 → 天然有序 → 可做范围查询
4. 最坏情况退化成链表 → 需要**平衡**（下一章）

---

## 2. 核心知识点

### 2.1 BST 定义
BST 不是随便排的二叉树——它有一条**全局约束**：对任何节点，**整个左子树都比它小，整个右子树都比它大**。这个约束直接决定了它的所有性质。

对每个节点 v：
- 左子树所有键 < v.key
- 右子树所有键 > v.key

→ **中序遍历产生升序序列**

![[bst_property_visualization.svg]]

### 2.2 基本操作

| 操作      | 思路                 | 复杂度  |
| ------- | ------------------ | ---- |
| Search  | 比较 key，递归去左或右      | O(h) |
| Insert  | 先 search 到空位，挂上新节点 | O(h) |
| Delete  | 三种情况处理             | O(h) |
| Min/Max | 一直往左/右             | O(h) |

其中 h 是树高，平衡 BST 时 h = O(log n)，退化时 h = O(n)。

**查找的精髓在于"二分"**——每次比较都能排除掉一整个子树。进入左子树时，右子树的全部节点被排除；进入右子树时，左子树的全部节点被排除。一棵高度为 h 的树最多走 h 步就能到底，所以查找是 O(h)。**插入**逻辑几乎一样——按查找规则走到 null 位置，把新节点挂在那里，也是 O(h)。

### 2.3 删除的三种情况
删除是 BST 操作里**最难**的一个。难点在于：删了节点之后，必须保证树**仍然是合法的 BST**。所以分三种情况处理：


1. **叶节点**：直接删除
2. **一个子节点**：用子节点替代
3. **两个子节点**：用**中序后继**（右子树最小）替换，再递归删除后继

![[bst_deletion.png]]

> 💡 想点按钮看三种情况的动态演示？👉 [点这里在浏览器打开](bst_deletion_three_cases.html)

**情况 3 最值得理解透：为什么必须用中序后继，不能随便找个值替换？**

被删节点在中序序列里有一个确定的位置，它的邻居就是中序前驱和后继。只有用这两个值之一替换，新的中序序列才仍然有序，BST 性质才不被破坏。中序后继有一个很好的结构特性——它是**右子树里最左的节点**，所以它最多只有一个右子节点（如果它还有左孩子，那它就不是右子树里最小的了）。因此递归删除它时，必然回到简单的情况 1 或 2，不会无限递归。

### 2.4 范围查询（Range Query）

- `findAll(k1, k2)`：找出所有 key ∈ [k1, k2] 的节点
- 思路：**剪枝搜索** —— 当前节点 key < k1 则只进右子树；key > k2 则只进左子树；在范围内则两边都进

这是 BST 比哈希表强的关键场景——哈希表查找 O(1) 但完全无序，**没法做范围查询**。BST 借助剪枝可以高效跳过整个无关子树，复杂度是 O(log n + s)（log n 找到边界，s 是结果数量），而不是遍历全部的 O(n)。

### 2.5 Index-Based Operations

给每个节点存**子树大小** size：
- `select(k)`：第 k 小元素，O(h)
- `rank(key)`：某 key 是第几小，O(h)

### 2.6 中序后继 / 前驱（Successor / Predecessor）

这两个概念在删除、范围查询、面试题里反复出现：

- **后继 = 右子树里最左的节点**（右子树存在时）；若无右子树，往上走找第一个"当前节点在其左子树中"的祖先
- **前驱 = 左子树里最右的节点**（左子树存在时）；若无左子树，对称往上走

直觉：后继必须比当前节点大且尽可能小 → 先去右子树（保证更大），再找右子树最小（最左节点）。

### 2.7 退化问题（Degeneration）

BST 最大的弱点：**插入顺序决定树的形状**。按有序顺序插入（1, 2, 3, 4...），每个新节点都挂到最右边，树退化成一条链表，所有操作从 O(log n) 变成 O(n)。

真正解决退化的方案是下一章的**平衡 BST**（AVL / 红黑树）：在每次插入/删除后做局部旋转，强制把高度压在 O(log n)，无论插入顺序如何。

### 2.8 Random BST

- 随机插入 n 个键的 BST 期望高度 = O(log n)
- 是后续 Treap、Skip List 的理论基础

## 3. 时间复杂度汇总

| 操作 | 平衡 | 最坏（退化）|
|---|---|---|
| Search / Insert / Delete | O(log n) | O(n) |
| Successor / Predecessor  | O(log n) | O(n) |
| Range Query (s 个结果)   | O(log n + s) | O(n) |

|结构|查找|插入|范围查询|有序性|
|---|---|---|---|---|
|有序数组|O(log n)|O(n)|强|有|
|链表|O(n)|O(1)*|弱|看实现|
|哈希表|O(1)|O(1)|**不支持**|**无**|
|**平衡 BST**|**O(log n)**|**O(log n)**|**O(log n + s)**|**有**|

BST 是**唯一同时拿到这四项的结构**。代价是常数因子比哈希表大，且需要保持平衡。这就是为什么 STL `std::map`、Java `TreeMap`、数据库索引（B+ 树是 BST 的多叉版本）都选它——只要你需要"有序"或"范围查询"，BST 家族就是默认答案。

**"什么时候会用到 BST？"**

判断标准只有一条：**你需要"有序 + 动态"同时成立吗？**

|场景|为什么 BST？|
|---|---|
|**联系人按姓名排序**，同时频繁增删|有序数组插入慢 O(n)；哈希无序；BST 两者都是 O(log n)|
|**数据库范围查询** `WHERE price BETWEEN 100 AND 500`|哈希做不到范围查询；BST 中序剪枝直接出结果|
|**排行榜** 实时插入分数 + 查"第 k 名是谁"|用 size 字段的 BST，`select(k)` 是 O(log n)|
|**文件系统目录树**|目录名有序，查找和插入都需要快|

**反过来，不该用 BST 的情况：**

- 只查不改 → 用有序数组 + 二分，Cache 更友好
- 不需要有序 → 用哈希表，O(1) 更快
- 数据可能顺序插入（会退化） → 直接用平衡树（下一章的 AVL / 红黑树）

**你现在用的工具里，BST 无处不在：**

- Python 的 `sortedcontainers.SortedList` → 背后是平衡 BST
- Java `TreeMap` / C++ `std::map` → 红黑树（BST 的平衡版）
- 任何数据库的 B-Tree 索引 → BST 的多叉推广

## 4. 常考陷阱

- ❌ 删除两个子节点的节点时，不能随便找个替代 —— 必须中序前驱/后继
- ❌ 顺序插入（1,2,3,...）会退化成链表
- ❌ BST 不允许重复 key（或需约定左/右放）
- ❌ 判断是否 BST 时，"左子树 < 根 < 右子树" 是**全局约束**，不是仅看孩子

---

## 5. 面试高频题

1. **验证一棵树是否 BST** —— 中序遍历判严格递增；或递归传 (min, max) 区间
2. **BST 的第 k 小元素** —— 中序遍历到第 k 个；或用 size 字段 O(log n)
3. **两个节点的最低公共祖先（LCA）** —— 第一个位于两键之间的节点
4. **BST 转双向链表** —— 中序遍历时改指针

## 6. 三视角职业分析

- **工程师视角**：STL `std::map`、Java `TreeMap` 底层是红黑树（BST 变体）
- **研究员视角**：BST 是"平衡树家族"的理论起点
- **系统架构师视角**：数据库索引底层的 B+Tree 是 BST 的多叉扩展
