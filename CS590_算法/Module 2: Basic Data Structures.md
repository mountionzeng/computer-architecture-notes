## 1. 思考脉络

1. 最简单的需求：按顺序存一堆东西 → **数组**
2. 数组插入/删除慢 → **链表**
3. 只从一端进出 → **栈**（调用栈、撤销）
4. 一端进一端出 → **队列**（BFS、任务调度）
5. 数据有层次关系 → **树**
6. 怎么访问树上所有节点 → **遍历**（DFS/BFS）

## 2. 核心知识点

### 2.1 栈（Stack）—— LIFO
- Last In, First Out
- 接口：`push`, `pop`, `top`, `isEmpty`
- 栈和队列的差别只在**进出方向**
- 实现：数组（需扩容）/ 链表
- 应用：函数调用栈、括号匹配、表达式求值、DFS

### 2.2 队列（Queue）—— FIFO
- First In, First Out
- 接口：`enqueue`, `dequeue`, `front`
- 实现：**循环数组**（避免搬移）/ 链表
- 变种：Deque（双端队列）

### 2.3 链表（Linked List）
数组的痛点是**插入和删除要搬移大量元素**。比如在数组中间插入一个元素，后面所有元素都要往后挪一格。链表用"指针"解决了这个问题——每个节点不需要在内存里相邻，而是通过指针告诉下一个节点在哪。

| 类型   | 指针          | 特点                                                          |
| ---- | ----------- | ----------------------------------------------------------- |
| 单链表  | next        | 最简单，只能顺着 next 一直往前走，回不了头。最省内存（每节点只多 1 个指针），但要找前驱节点必须从头开始遍历。 |
| 双链表  | prev + next | 每个节点同时记着"前一个"和"后一个"。代价是每个节点多用一个指针的内存，但删除当前节点、反向遍历都变得很容易。    |
| 循环链表 | 尾指向头        | 尾节点的 next 不是 null，而是指回头节点。适合做"环形缓冲区"或者轮询调度（比如多人轮流发牌）。       |

#### 链表 vs 数组：插入操作的本质区别
数组插入要做"挪车位"——后面的车都得往后挪一格，所以是 O(n)。链表插入只要做"改指路牌"——把 B 的 next 从原来指向 C 改成指向 X，再让 X 指向 C，两步搞定，O(1)。

但这里有个**关键陷阱**——你笔记里写的"链表插入是 O(1)*"那个星号说的就是这个：

链表"插入操作本身"是 O(1)，但前提是**你已经站在 B 节点上**。如果你只知道"在第 1000 个位置插入"，那要先从头走到第 1000 个节点，这一步是 O(n)。所以：

- 如果你已经持有目标节点的引用 → O(1)
- 如果只知道索引或值 → 整体还是 O(n)

这就是为什么数组的"按索引访问"是 O(1)（直接算地址 `base + i × size`），而链表的访问是 O(n)（必须从头沿指针走）。**两种结构在不同操作上是镜像优劣**。

### 2.4 树（Tree）基础

- **定义**：无环连通图，n 个节点 → n-1 条边
- **术语**：root, parent, child, sibling, leaf, depth, height
- **二叉树特例**：每节点 ≤ 2 子节点
- **满/完全/完美**二叉树的区分

### 2.5 树的遍历
树和前面的线性结构完全不同——它表达的是**层次关系**，比如文件夹嵌套、组织架构、表达式结构。下面这棵树是同一棵，但用不同顺序遍历会产生不同序列：

这里的 DFS / BFS 会在图遍历中反复出现：**DFS = Depth-First Search（深度优先搜索）**，常用栈或递归；**BFS = Breadth-First Search（广度优先搜索）**，常用队列。

| 顺序          | 中文  | 规则                       | 应用                                           |
| ----------- | --- | ------------------------ | -------------------------------------------- |
| Pre-order   | 前序  | 根 → 左 → 右，DFS （用栈或递归实现）  | 复制一棵树（先建根，再递归建子树）                            |
| In-order    | 中序  | 左 → 根 → 右 ，DFS （用栈或递归实现） | BST 升序，在二叉搜索树（BST）上**输出有序序列**——这是 BST 最重要的性质 |
| Post-order  | 后序  | 左 → 右 → 根，DFS （用栈或递归实现）  | 删除一棵树（先删子树，再删根，避免父节点丢失子节点引用）；表达式树求值          |
| Level-order | 层序  | 按层（BFS）用队列实现             | 最短层，找最近的元素、按层处理                              |

## 3. 操作复杂度对比

| 结构 | 访问 | 搜索 | 插入 | 删除 |
|---|---|---|---|---|
| 数组 | O(1) | O(n) | O(n) | O(n) |
| 链表 | O(n) | O(n) | O(1)* | O(1)* |
| 栈   | — | — | O(1) | O(1) |
| 队列 | — | — | O(1) | O(1) |

*链表插入删除本身 O(1)，但找到位置是 O(n)

### 3.1 查找方式速查：数组 / 有序数组 / 哈希表

| 场景 | 前提 | 查找方式 | 时间复杂度 |
|---|---|---|---|
| 普通未排序数组里找某个值 | 没有排序信息 | 从头到尾线性扫描 | O(n) |
| 有序数组里找某个值 | 数组已排序 | 二分查找，每次排除一半 | O(log n) |
| 哈希表按 key 找 value | 哈希函数分布较均匀、冲突不严重 | `key -> hash function -> 位置` | 平均 O(1)，最坏 O(n) |

> 记忆钩子：**没顺序就线性扫，有顺序就二分，有 key 映射就哈希直达**。

**最关键的直觉**是这一点：**没有"最好"的数据结构，只有"最适合当前操作模式"的数据结构。**

如果你的程序大量做"按下标读取"（比如图像像素），数组完胜；如果大量做"中间插入删除"（比如文本编辑器的字符流），链表更优；如果只在两端操作并要求严格顺序（撤销栈、任务队列），就用栈或队列把接口"封住"，反而能避免误用。

这种"用接口约束来防止错误"的思想，是栈和队列存在的真正意义——它们其实可以用数组或链表实现，但**故意只暴露 push/pop 或 enqueue/dequeue**，让代码意图更清晰。

---

整个 Module 2 的核心可以归纳为一句话：**数组和链表是两种基础的"摆放方式"，栈、队列、树都是在它们之上构建的更高层抽象**。后面学的哈希表、堆、图，本质都是这两种摆放方式的进一步组合。

## 4. 常考陷阱

- ❌ 循环队列要区分"空"和"满"：常用 `size` 或留一格
- ❌ 单链表删除需要"前驱"指针，删头节点要特判
- ❌ 递归遍历深度过大可能爆栈 → 改迭代 + 显式栈
- ❌ 完全二叉树 ≠ 满二叉树

## 6. 三视角职业分析

- **工程师视角**：80% 业务代码就是 List/Map/Stack 的排列组合
